**版本**：v1.0

**状态**：待审查

## 1. 背景与问题

### 1.1 当前核心痛点

| # | 痛点 | 具体表现 | 后果 |
|---|------|---------|------|
| 1 | **Agent 信号无因子依据** | 4个分析师直接调用 AKShare 原始数据，无统一特征标准化，数据质量参差不齐 | 不同标的、不同时间的信号可比性差 |
| 2 | **调仓建议用随机数** | `pipeline.py:87` `random.uniform(0.03, 0.08)` 决定调仓幅度 | 相同信号给出不同建议，审批人无从判断合理性 |
| 3 | **没有因子质量评估** | 平台无法告诉用户"这个技术指标真的有预测力吗" | Agent 输出有无参考价值不可知 |
| 4 | **回测是假的** | `services/backtest.py` 使用 GBM 随机模拟，与 Agent 历史信号完全无关 | 回测结果零参考价值，无法验证策略有效性 |
| 5 | **Agent 权重永远固定** | `pipeline.py` 中各 Agent 权重硬编码，不随历史表现更新 | 表现差的 Agent 与表现好的 Agent 影响力相同 |
| 6 | **无因子自动发现能力** | 研究员只能手动尝试指标，无系统化的因子挖掘流程 | 平台对新因子机会响应慢，依赖人工经验 |

### 1.2 已有可用基础设施

| 模块 | 文件 | 可复用能力 |
|------|------|----------|
| AKShare 接入 | `services/cn_market_data.py` 等 | 已有技术/基本面/新闻/情绪数据拉取，统一异步包装 |
| 规则引擎 | `agents/cn_technical_agent.py` | RSI/MACD/布林带等技术指标计算已实现 |
| 回测框架壳 | `services/backtest.py` `routers/backtest.py` | 接口、手续费/滑点模型、净值计算框架已有 |
| 决策历史 | `models.py:DecisionRun` | 完整记录了每次分析的 agent_signals 和 final_direction |
| Agent Pipeline | `agents/pipeline.py` | 4 Agent 并发 + 加权投票结构清晰，便于注入因子上下文 |
| WebSocket | `websocket_manager.py` | 长任务进度推送基础设施已就绪 |

### 1.3 外部参考

本版本核心设计借鉴两个开源项目的思想：

- **karpathy/autoresearch**：自主研究循环——固定计算预算内反复「生成假设 → 评估 → 保留/丢弃」，无需人工干预每次实验
- **QuantaAlpha/QuantaAlpha**：LLM 驱动因子挖掘——自然语言描述研究方向 → 多样化因子假设 → 代码生成 → IC 回测验证 → 进化优化，在 CSI 300 验证超额 27.75%/年

## 2. 目标

| 目标 | 描述 |
|------|------|
| **主目标 1** | 建立因子基础设施：特征标准化 → IC 评估引擎 → 因子库，使平台具备因子质量评估能力 |
| **主目标 2** | 实现自主因子挖掘：研究员输入研究方向，系统自动生成假设、评估 IC、筛选高质量因子入库 |
| **主目标 3** | Agent 接入因子层：Pipeline 从因子库读取经过验证的因子值注入各 Agent，调仓建议去除随机数 |
| **主目标 4** | 回测真实化：回测引擎改为基于历史 Agent 信号驱动，准确率可追踪，权重动态调整 |
| **约束** | 不引入 GPU 依赖；不修改现有审批/风控接口；因子计算沙盒不能访问外部网络；AKShare 调用保持 `asyncio.to_thread()` 封装 |

## 3. 用户故事

### US-01：特征数据标准化查询

> As a **量化研究员**,
> I want **通过统一 API 查询任意标的在任意日期的所有特征值（RSI、PE、北向资金等），并知道每个特征的数据质量和来源**,
> So that **构建因子时有干净、可重复的数据基础，不需要自己处理 AKShare 的各种格式差异和缺失值**。

### US-02：手动创建因子并评估 IC

> As a **量化研究员**,
> I want **在因子库页面手动输入一个因子表达式（如"近20日收益率除以波动率"），系统自动计算过去 60 个交易日的 IC 序列并给出 IC_IR**,
> So that **我可以快速验证一个因子假设是否有统计意义，不需要写脚本**。

### US-03：启动自主因子挖掘任务

> As a **量化研究员**,
> I want **输入一段自然语言研究方向（如"融资余额与股价的关系"），系统自动生成多个因子假设、逐一评估、筛选 IC_IR 达标的因子进入因子库，并在前端看到实时进度**,
> So that **我可以快速探索一个新的研究方向，不需要手动一个个试因子**。

### US-04：因子库管理与监控

> As a **量化研究员**,
> I want **在因子库页面看到所有 active 因子的质量指标（IC 均值、IC_IR、覆盖率），以及 IC_IR 随时间的变化趋势，衰减的因子自动标橙色警告**,
> So that **我知道哪些因子当前可信、哪些已经失效，避免使用过时的因子**。

### US-05：Agent 分析引用因子得分

> As a **审批人**,
> I want **在决策详情页看到"综合因子得分：0.72（偏多）"以及各类因子的具体数值和 IC_IR，而不只是 Agent 的文字推理**,
> So that **我能判断 Agent 的方向判断是否有量化数据支撑，不只是 LLM 的文字描述**。

### US-06：调仓建议有确定性依据

> As a **审批人**,
> I want **推荐仓位的变动幅度（如"建议从 5% 增至 8.6%"）是基于因子得分和 Agent 置信度计算的确定性结果，相同输入每次给出相同建议**,
> So that **我能理解调仓幅度的来源，不会出现相同分析结果给出不同仓位建议的混乱情况**。

### US-07：基于真实历史信号的回测

> As a **量化研究员**,
> I want **选择一段时间范围，系统基于这段时间内真实的 Agent 信号和 T+1 开盘价模拟执行，生成净值曲线，并能追溯每次涨跌对应哪个具体决策**,
> So that **我能真实评估历史 Agent 信号的价值，而不是随机模拟出来的假曲线**。

### US-08：Agent 准确率追踪与权重动态调整

> As a **系统**,
> I want **每笔已完成的信号在 T+5 交易日自动拉取实际价格、计算各 Agent 预测准确率，并每 20 个交易日按近 60 天准确率更新各 Agent 权重**,
> So that **表现好的 Agent 自动获得更高权重，表现差的 Agent 被自动降权，系统具备自我优化能力**。

## 4. 验收标准（EARS 格式）

### AC-01：特征数据标准化接入

```
GIVEN 研究员调用 GET /api/v1/features/600036?date=2026-04-01
WHEN 请求成功返回
THEN 响应包含各类特征值（至少覆盖技术/基本面/情绪三类）
AND  每个特征包含 value、raw_value、source、coverage、updated_at 字段
AND  缺失特征用行业中位数填充，并在 fill_method 字段标注 "industry_median"
AND  响应时间 < 2s（数据已缓存时）
```

### AC-02：特征值标准化正确性

```
GIVEN 技术特征 rsi_14 的原始值为 72.3
WHEN 系统计算标准化特征
THEN z_score = (72.3 - 全市场同日 RSI 中位数) / MAD，精度 ≤ 0.001
AND  经过 winsorize(1%, 99%) 极值处理后再计算
```

### AC-03：手动因子 IC 评估

```
GIVEN 研究员在因子库页面创建因子，表达式为 "rsi_14"，forward_days=5
WHEN 点击「评估 IC」，后端获取过去 60 个交易日数据
THEN 系统计算并展示 IC 均值、IC_IR、IC 为正比例、IC 时间序列折线图
AND  计算结果与手动 Excel Spearman 相关系数计算偏差 < 0.001
AND  评估完成时间 < 30s
```

### AC-04：防止未来数据泄漏

```
GIVEN IC 评估使用截面日 T 的因子值和 T+N 的未来收益
WHEN 系统计算 IC 序列
THEN 因子值只使用截面日 T 当天或之前的数据
AND  未来收益使用 T+1 开盘价起算的真实收益
AND  若因子数据日期晚于截面日，该截面跳过并记录 skip_reason="data_leak_risk"
```

### AC-05：自主因子挖掘 — 任务启动

```
GIVEN 研究员在挖掘页面输入研究方向文字，点击「启动挖掘」
WHEN 任务创建成功
THEN 后端创建 FactorMiningTask 记录，status=running
AND  前端通过 WebSocket 实时接收迭代进度（当前迭代数 / 总迭代数）
AND  页面显示每次实验的假设名称、IC 值、结论（入库/淘汰/原因）
```

### AC-06：自主因子挖掘 — 因子筛选标准

```
GIVEN 一轮挖掘实验完成评估
WHEN 判断是否入库
THEN IC_IR ≥ 0.5 且 与现有 active 因子的 Spearman 相关性 < 0.6 → 入库（status=active）
AND  IC_IR 0.3~0.5 → 存入因子库（status=watchlist），不参与 Agent 注入
AND  IC_IR < 0.3 → 丢弃，记录原因到实验日志
AND  因子代码执行超时（> 30s）或抛出异常 → 丢弃，记录 discard_reason="execution_error"
```

### AC-07：LLM 生成因子多样性

```
GIVEN 研究员输入同一研究方向
WHEN LLM 生成一批因子假设（通常 5~8 个）
THEN 假设至少覆盖 3 种不同维度（绝对值/Z-Score/变化率/排名 等）
AND  LLM 输出格式为 JSON 数组，每个元素包含 name、expression、features_needed、direction、rationale
AND  LLM 输出格式校验失败时，任务标记 warning 但不中断，跳过该假设继续下一个
```

### AC-08：因子沙盒安全执行

```
GIVEN LLM 生成的因子 Python 代码
WHEN 在沙盒中执行
THEN 只允许 pandas / numpy 内置函数，import 其他模块抛出 NameError
AND  执行超时 30s 强制中止
AND  代码执行产生的异常不影响主进程，异常信息记录到实验日志
```

### AC-09：Agent 接收因子上下文

```
GIVEN Pipeline 调用各 Agent 分析前，因子库中有 ≥ 1 个 active 因子
WHEN Pipeline 调用 build_factor_context(symbols)
THEN 返回包含各标的因子值、z_score、ic_ir 和 factor_composite_score 的字典
AND  factor_composite_score 为各 active 因子按 IC_IR 加权的综合得分，范围 [0, 1]
AND  Agent Prompt 包含因子数值（如 "综合因子得分: 0.72，技术因子: RSI=68.5 z=1.2"）
AND  因子库为空时，context["factor_context"] = {}，Agent 正常运行不报错
```

### AC-10：调仓建议去除随机数

```
GIVEN Pipeline 完成投票，final_direction=buy，factor_composite_score=0.72，avg_conf=0.75
WHEN 调用 _make_recommendations
THEN weight_delta = max_delta × |factor_score - 0.5| × 2 × avg_conf（确定性公式）
AND  相同输入连续调用 10 次，每次 recommended_weight 结果完全一致
AND  不再调用 random.uniform
```

### AC-11：真实历史信号回测

```
GIVEN 数据库中有 ≥ 10 条 status=completed 的 DecisionRun
WHEN 研究员触发回测，选择时间范围
THEN 回测引擎按 started_at 排序历史信号，不使用 GBM 模拟
AND  每条信号的执行价格为 started_at 当日后第一个交易日的开盘价（T+1）
AND  净值曲线中每次涨跌可追溯到具体 DecisionRun.id
AND  历史信号不足 10 条时顶部显示警告 "历史信号不足，结果参考价值有限"
```

### AC-12：防止回测未来数据泄漏

```
GIVEN 回测引擎执行历史信号序列
WHEN 检测到某条信号的 started_at 晚于其执行日期（违规）
THEN 该信号跳过，记录 skip_reason="future_leak"
AND  跳过数量显示在回测报告顶部警告区
AND  回测期间不重新调用 LLM（LLM 知道历史必然泄漏）
```

### AC-13：Agent 准确率自动结算

```
GIVEN 某条 DecisionRun 已完成（status=completed），且距 started_at ≥ 5 个交易日
WHEN 定时任务触发结算
THEN 后端拉取 T+5 真实价格，与 final_direction 对比，写入 AgentAccuracyRecord
AND  各 Agent 的 predicted_direction 与实际涨跌方向一致计为 correct
AND  累积 15 条以上已结算记录后，Dashboard Agent 卡片显示真实准确率
```

### AC-14：Agent 权重动态调整

```
GIVEN 每 20 个交易日触发一次权重更新
WHEN 计算各 Agent 近 60 天准确率
THEN 准确率 < 45% → 权重上限 10%
AND  准确率 45%~55% → 维持基础权重
AND  准确率 > 65% → 权重可提升至 35%
AND  所有权重归一化为 1.0 后写入 AgentWeightConfig 表
AND  Pipeline 运行时从数据库读取最新权重，不再硬编码
```

### AC-15：因子衰减监控

```
GIVEN 某 active 因子距上次评估已过 20 个交易日
WHEN 系统触发定期重评估
THEN 重新计算近 60 天 IC_IR
AND  IC_IR 下降超过 40%（相对原始入库时）→ status=degraded，因子卡片标橙色
AND  IC_IR < 0.1 → status=archived，不再注入 Agent
```

## 5. 非功能性需求

| 类别 | 要求 |
|------|------|
| **性能** | 特征查询（已缓存）< 2s；IC 评估（60日窗口）< 30s；因子上下文注入（全部 active 因子）< 5s |
| **安全** | 因子沙盒只允许 pandas/numpy，禁止 import sys/os/subprocess 等；禁止网络访问 |
| **数据质量** | 覆盖率 < 50% 的特征不参与 IC 评估，记录 skip_reason="low_coverage" |
| **防泄漏** | IC 评估、回测均强制检查时间戳前序性，违规数据自动跳过并记录 |
| **兼容性** | 不修改现有 `decisions`、`approvals`、`portfolio`、`risk` 接口；新功能均为新增路由和表 |
| **Mock 模式** | LLM_API_KEY 为空时，因子挖掘的假设生成使用内置 Mock 假设列表（3~5 个预设），不报错 |
| **AKShare 规范** | 所有 AKShare 调用必须用 `asyncio.to_thread()` 封装；接口名做多接口兜底 |

## 6. 超出范围（Out of Scope）

| 功能 | 不做的原因 |
|------|----------|
| Tick 级 / 分钟级特征 | 日线级别够用，分钟级需专业数据服务且成本高 |
| GPU 加速 / 深度学习模型 | 因子计算是 CPU 密集型，IC 评估无需 GPU |
| 强化学习自动选因子 | IC + 进化算法更透明可解释，RL 黑盒在数据量不足时容易过拟合 |
| 因子中性化（市值/行业回归残差） | 高级功能，先验证基础 IC 框架有效性 |
| 多因子选股回测（构建投资组合） | v12.0 规划内容，依赖因子库充分积累后再做 |
| 因子商业数据（Wind/Bloomberg） | 优先用 AKShare 免费数据验证框架，商业数据接入单独立项 |
| 跨版本权重锁定（UI） | 本版本权重更新是自动的；手动锁定某 Agent 权重的 UI 留到 v5.0 规则管理 |

```python
# 1. 特征快照缓存
class FeatureSnapshot(Base):
    symbol: str         # 标的
    date: str           # 日期 YYYY-MM-DD
    features: str       # JSON，所有特征的 {name: {value, z_score, source, coverage}}
    updated_at: datetime

# 2. 因子定义
class FactorDefinition(Base):
    id: str             # 因子唯一标识（slug）
    name: str
    description: str
    expression: str     # Python 表达式
    factor_type: str    # technical/fundamental/sentiment/composite
    source_features: str  # 依赖的特征名列表（JSON）
    created_by: str     # human/llm/evolved
    status: str         # active/watchlist/archived/degraded
    hypothesis: str     # 原始研究假设（LLM 挖掘时填入）
    created_at: datetime

# 3. 因子评估记录
class FactorEvaluation(Base):
    factor_id: str      # FK → FactorDefinition
    eval_start_date: str
    eval_end_date: str
    forward_days: int   # 预测窗口
    ic_mean: float
    ic_ir: float
    ic_positive_pct: float
    turnover_rate: float
    coverage_rate: float
    raw_result: str     # 完整评估结果 JSON
    evaluated_at: datetime

# 4. 因子挖掘任务
class FactorMiningTask(Base):
    task_name: str
    research_direction: str
    status: str         # running/completed/failed
    iterations: int     # 已完成迭代数
    best_factor_id: str
    best_ic_ir: float
    config: str         # 任务配置 JSON（预算/阈值等）
    started_at: datetime
    completed_at: datetime

# 5. Agent 准确率记录
class AgentAccuracyRecord(Base):
    decision_run_id: str  # FK → DecisionRun
    agent_type: str       # technical/fundamental/news/sentiment
    predicted_direction: str
    actual_direction: str  # T+5 实际涨跌方向
    is_correct: bool
    settled_at: datetime

# 6. Agent 权重配置（动态）
class AgentWeightConfig(Base):
    agent_type: str
    weight: float       # 当前权重（归一化后）
    accuracy_60d: float # 近60天准确率
    updated_at: datetime
    locked: bool        # 是否锁定（不参与自动调整）
```

### 7.3 IC 评估引擎核心逻辑

```python
# backend/services/factor_eval.py

async def evaluate_factor_ic(
    factor_id: str,
    eval_window_days: int = 60,
    forward_days: int = 5,
) -> dict:
    """
    1. 从 FeatureSnapshot 拉取 eval_window_days 个截面日的因子值
    2. 从 AKShare 拉取对应的 T+forward_days 真实收益率
    3. 对每个截面日计算 Spearman IC
    4. 返回 ic_mean / ic_ir / ic_positive_pct 等指标
    """
    ics = []
    for date in eval_dates:
        fv = get_factor_cross_section(factor_id, date)       # 全市场截面因子值
        fr = get_future_returns(date, forward_days)           # T+N 收益率
        common = fv.index.intersection(fr.index)
        if len(common) < 20:
            continue
        ic = fv[common].corr(fr[common], method='spearman')
        ics.append({"date": date, "ic": ic})

    ic_series = pd.Series([x["ic"] for x in ics])
    return {
        "ic_mean": ic_series.mean(),
        "ic_ir":   ic_series.mean() / ic_series.std() if ic_series.std() > 0 else 0,
        "ic_positive_pct": (ic_series > 0).mean(),
        "ic_series": ics,
    }
```

### 7.4 自主挖掘引擎流程

```python
# backend/services/factor_mining.py

async def run_mining_task(task_id: int, ws_broadcaster=None):
    """
    AutoResearch 式研究循环：
    1. LLM 生成 5~8 个多样化因子假设
    2. 对每个假设：生成代码 → 沙盒执行 → IC 评估 → 入库/淘汰
    3. 对 active 因子做进化变异（改变回看周期/加标准化/组合等）
    4. 重复直到达到 max_iterations 或 best_ic_ir > 1.0
    """
    # 进化变异策略
    MUTATION_STRATEGIES = [
        lambda expr: f"rolling_mean({expr}, 5)",           # 5日均化
        lambda expr: f"({expr} - rolling_mean({expr}, 60)) / rolling_std({expr}, 60)",  # Z-Score
        lambda expr: f"rank({expr})",                      # 排名因子
        lambda expr: f"({expr}) / rolling_std(price, 20)", # 波动率调整
    ]
```
