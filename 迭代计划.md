## 二、v2.0 — 让信号真正有依据

**优先级**：🔴 最高，不完成此版本后续版本无意义

> 核心原则：每一条 Agent 信号必须能追溯到具体数据，LLM 只做推理和摘要，不做数值计算。

---

### v2.1 技术分析师改为规则引擎

**问题**：LLM 对价格序列的数值推理很弱，同样的 RSI=72 可能给出截然相反的结论，且每次结果不一致。技术指标是确定性计算，不应该让 LLM 做。

**方案**：技术分析师的方向判断改为纯 Python 规则引擎，规则命中后 LLM 只负责把结果翻译成中文摘要。

规则优先级（从高到低）：
```python
if rsi > 75:                                → sell, confidence=0.82  # 超买
if rsi < 28:                                → buy,  confidence=0.80  # 超卖
if price > ma20 > ma50 and macd > signal:   → buy,  confidence=0.72  # 趋势确认看多
if price < ma20 < ma50 and macd < signal:   → sell, confidence=0.70  # 趋势确认看空
if price > bb_upper:                        → sell, confidence=0.65  # 突破布林上轨
if price < bb_lower:                        → buy,  confidence=0.63  # 突破布林下轨
default:                                    → hold, confidence=0.50
```

LLM 只做摘要翻译，不参与方向判断：`"RSI=72 处于超买区（>70），价格触及布林上轨，技术面偏空。"`

**验收**：相同输入 10 次分析，方向和置信度每次完全一致，推理摘要包含具体数值。

---

### v2.2 基本面分析师接真实财务数据

**问题**：当前靠 LLM 训练数据里的财务知识，数据可能是一两年前的，且存在幻觉风险。

**方案**：接入数据API，每次分析前拉取最新财务数据，7 天本地缓存。

拉取字段：PE 比率、收入增速（YoY）、净利润率、债务股权比、自由现金流、EPS

无法获取时自动 fallback，`data_sources` 标注 `"LLM知识库（无实时数据，仅供参考）"`。

**验收**：Agent 信号卡片数据来源显示 `"FMP 2025-Q4"`，推理摘要包含具体财务数字（如 `"PE=28.5，高于行业均值24.2"`）。

---

### v2.3 新闻分析师接真实新闻源

**问题**：新闻分析师完全靠 LLM 记忆，无法感知今天发生的事情。

**方案**：接入新闻数据API，每次分析前拉取近 24h 相关新闻，2 小时缓存

注入 Prompt 格式：
```
近24h新闻（3条）：
1. [Reuters 2026-03-20] AAPL beats Q1 earnings, EPS $2.18 vs expected $2.05
2. [Bloomberg 2026-03-20] Apple Vision Pro sales disappoint analysts
请基于以上新闻分析 AAPL 的短期催化剂...
```

无新闻时明确标注：`"近24h无重大新闻，基于历史背景和宏观环境分析"`

**验收**：新闻分析师信号卡片的 `data_sources` 显示具体新闻标题、来源和时间戳。

---

### v2.4 情绪分析师接恐慌贪婪指数

**问题**：情绪分析师只用了 RSI 做代理指标，缺少市场整体情绪数据。

**方案**：接入对应数据API，结合个股 RSI 综合判断。

情绪映射（逆向指标）：
- 指数 0-25（极度恐慌）→ 偏多，confidence=0.72
- 指数 25-45（恐慌）→ 轻度多，confidence=0.60
- 指数 55-75（贪婪）→ 轻度空，confidence=0.60
- 指数 75-100（极度贪婪）→ 偏空，confidence=0.72

**验收**：情绪分析师信号卡片展示真实恐慌贪婪指数数值，可与权威计算结果对比验证。

---

### v2.5 每条信号绑定完整数据快照

**问题**：当前信号只有结论，无法追溯分析时用了什么数据、数据是什么时候采集的。

**验收**：决策详情展开技术分析师，能看到分析时的 RSI、MACD 等具体数值和采集时间。

## 三、v3.0 — 让回测真正有意义

**优先级**：🔴 高，依赖 v2.0 完成

---

### v3.1 回测引擎改为基于真实历史信号

**问题**：当前回测是独立随机 GBM，和 Agent 完全无关，结果零参考价值。

**核心设计原则**——防止未来数据泄漏：
```
✅ 允许：用 T 日 started_at 之前的数据生成信号
✅ 允许：在 T+1 开盘价执行（模拟操作延迟）
❌ 禁止：信号的 started_at 晚于执行日期
❌ 禁止：回测时重新调用 LLM（LLM 知道历史必然泄漏）
```

主要流程：
1. 筛选时间范围内已完成的 `DecisionRun`，按 `started_at` 排序
2. 验证每条信号时间戳前序性，违规跳过并记录
3. 拉取 T+1 真实开盘价作为执行价格
4. 按 `final_direction` 驱动仓位变化，扣除手续费和滑点
5. 同步计算买持基准净值用于对比

历史信号不足 10 条时顶部显示警告，不足时 fallback 到 GBM 模拟并标注"模拟数据，仅供参考"。

**验收**：完成 15 次以上分析后触发回测，净值曲线的每次涨跌可追溯到具体 `DecisionRun` 记录。

---

### v3.2 Agent 历史准确率追踪

**问题**：不知道每个 Agent 的历史准确率，无法判断哪个 Agent 更可信。

**方案**：分析完成后第 5 个交易日，拉取实际价格，计算各 Agent 预测方向是否正确。

**验收**：累积 15 条以上已结算记录后，Dashboard Agent 卡片显示真实准确率，可追溯到具体决策。

---

### v3.3 Agent 信号权重动态调整

**问题**：4 个分析师权重固定，不会根据历史表现自动优化。

**方案**：每 20 个交易日，用各 Agent 近 60 天准确率重新计算权重。

```python
# 权重计算规则
准确率 < 45%  → 权重上限 10%（弱信号保护）
准确率 45-55% → 维持基础权重
准确率 > 65%  → 权重可提升至 35%（强信号奖励）
所有权重归一化为 1.0
```

权重更新写入 `AgentWeightConfig` 表，pipeline 运行时从数据库读取，不再硬编码。

**验收**：某 Agent 近 60 天准确率 65%，权重自动提升；可在规则页锁定某 Agent 权重不被自动调整。

---

### v3.4 回测报告导出

**前端改动**：

- 回测完成后右上角增加「导出 JSON」按钮
- 增加「复制配置」按钮，将当前参数填回表单方便微调重跑

## 四、v4.0 — 让风控真实有效

**优先级**：🔴 高，依赖 v2.0 完成

---

### v4.1 真实持仓追踪

**问题**：系统不追踪真实持仓，风控检查基于 Agent 建议值而非实际持仓。

**前端改动**：

- `DashboardPage` 新增持仓看板：各标的当前权重、成本价、浮动盈亏（yfinance 实时计算）
- `AnalyzePage` 当前持仓输入框变为只读自动填入，保留手动覆盖功能

**验收**：审批通过 AAPL buy 后，Dashboard 出现 AAPL 持仓记录；再次触发分析，`current_weight` 自动读取真实持仓。

---

### v4.2 仓位上限在审批层强制执行

**问题**：仓位上限只是软建议，审批时不做二次校验，实际可以绕过。

**验收**：把仓位上限改为 5%，审批建议 AAPL 权重 10% 的记录，后端返回 422，前端显示具体原因。

---

### v4.3 日内净值自动监控与熔断触发

**问题**：三级熔断的阈值配好了，但没有机制自动计算净值变化，熔断永远不会自动触发。

**前端改动**：
- `DashboardPage` 持仓看板增加"当日净值变化"进度条（接近警告线变橙，超过变红）
- 熔断触发时 WebSocket 推送，顶部出现红色横幅，点击跳转规则页

**验收**：临时把触发阈值改为 0.001%，观察到监控任务自动触发熔断，Dashboard 出现红色横幅。

---

### v4.4 多标的组合级视图

多标的时增加组合汇总步骤

```python
portfolio_summary = {
    "net_buy_exposure": ...,   # 买入方向标的权重之和
    "concentration_risk": ..., # 最大单标的建议权重
    "symbols_count": len(symbols),
}
```

**前端改动**：

- `AnalyzePage` 多标的完成后，结果区顶部显示组合摘要卡片
- `DashboardPage` 持仓看板下方增加组合暴露进度条

## 五、v5.0 — 系统自我优化

**优先级**：🟡 中

---

### v5.1 自动审批规则真正生效

**问题**：创建审批记录时完全没检查自动审批规则，配置了也不执行。pipeline 完成后调用规则引擎，命中则设为 `auto_approved`，`trigger_count` +1

**验收**：创建"置信度 ≥ 0.8 自动批准"规则，触发高置信度分析，审批状态自动变为 `auto_approved`，无需人工操作。

---

### v5.2 提示词模板管理

**问题**：各 Agent 的 LLM prompt 硬编码在代码里，研究员无法调整分析侧重点。

pipeline分析前从数据库查询活跃模板，覆盖硬编码 prompt

支持变量占位符：`{symbol}`、`{price_summary}`、`{rsi}`、`{market_context}`

**前端改动**：`RulesPage` 增加"提示词管理" Tab，支持查看 / 编辑 / 切换各 Agent 的 prompt 版本

**验收**：修改技术分析师 prompt 增加量价背离关注，推理摘要出现量价背离相关描述。

---

### v5.3 信号逻辑一致性检测

**问题**：LLM 可能输出"推理说看空但 direction=buy"的自相矛盾信号。

增加校验：

```python
bearish_kw = ["超买", "overbought", "下跌风险", "bearish", "卖出压力", "回调"]
bullish_kw = ["超卖", "oversold", "上涨", "bullish", "买入机会", "支撑"]

if direction == "buy" and any(kw in reasoning for kw in bearish_kw):
    hallucination_events.append({
        "event_type": "logic_contradiction",
        "description": "方向(buy)与推理摘要存在矛盾"
    })
    # 只记录不修改方向，人工审批时参考
```

**前端改动**：出现矛盾的 Agent 信号卡片显示橙色矛盾警告标识

**验收**：构造矛盾信号，决策详情页对应卡片出现橙色警告，幻觉事件写入审计记录。

---

### v5.4 分析历史页与快速重分析

**前端改动**：
- `AnalyzePage` 底部增加"历史分析记录"折叠面板（最近 10 条）
- 每行有「查看详情」和「重新分析」按钮，「重新分析」自动填入标的代码

## 六、v6.0 — 用户体验完善

**优先级**：🟡 中

---

### v6.1 轻量用户认证

**问题**：审批人靠手填姓名，无身份验证，任何人可用任何名字操作。

**方案**：PIN 码 + JWT Token，24h 有效，不引入复杂 OAuth。

**默认用户**（首次启动打印到日志）：
- `admin` / PIN: `1234`（全部功能含风控修改）
- `analyst` / PIN: `5678`（触发分析和审批，不可改风控参数）

**前端改动**：
- 新增 `LoginPage.vue`（`/login`），PIN 码数字键盘
- 路由守卫：未登录跳转 `/login`
- Sidebar 底部显示当前用户名和登出按钮
- 审批操作移除手填姓名，自动使用登录用户名

**验收**：未登录访问 `/approvals` 自动跳转登录；analyst 角色无法修改风控参数（返回 403）。

---

### v6.2 通知与待办中心

**问题**：有新待审批或熔断触发时，用户需要主动刷新才能知道。

**补充 WebSocket 事件**：`backtest_completed`、`auto_approval_triggered`、`settlement_completed`、`risk_threshold_warning`

通知点击跳转：新待审批→`/approvals/{id}`、回测完成→`/backtest`、熔断触发→`/rules`、分析完成→`/decisions/{id}`

**验收**：分析完成后铃铛出现红点，点击显示"AAPL 分析完成，方向：买入"，点击跳转决策详情。

---

### v6.3 分析定时调度

**问题**：只能手动触发分析，研究员希望每天开盘前自动跑。

**新增模型**：`ScheduledTask`（id、symbols、cron_expr、is_active、last_run_at、next_run_at）

**新增文件**：`backend/services/scheduler.py`（基于 asyncio，支持 `0 9 * * 1-5` 格式）

**新增接口**：`/api/v1/scheduler/` CRUD

**前端改动**：`AnalyzePage` 触发表单下方增加"设置定时分析"折叠区域，快捷配置：每天 09:00 / 15:00 / 仅工作日

**验收**：创建"AAPL 每天 09:00 分析"任务，等待触发，系统自动创建 DecisionRun 并推送通知。

## 八、v8.0 — 执行自动化

**优先级**：🔵 长期 | **前提**：v2.0–v4.0 全部完成，且策略经过至少 6 个月纸面交易验证

---

### v8.1 纸面交易模式

审批通过后模拟执行，用真实价格计算盈亏，但不动真实资金。

`Portfolio` 表增加 `is_paper_trade` 标记，每天自动结算当日收益并写入净值历史。

运行 30 天后，Dashboard 展示基于真实价格和审批记录计算的策略净值曲线。

---

### v8.2 券商 API 接入

| 券商 | 适用场景 | 接入难度 |
|------|---------|---------|
| Interactive Brokers（IBKR） | 全球市场，专业用户 | 中等 |
| 富途 OpenAPI | 港美股，有中文文档 | 较低 |
| 老虎证券 OpenAPI | 港美股 | 较低 |
| 其他 | 中国市场 |  |

接入后流程：
```
审批通过 → 后端调用券商 API 下单 → 获取成交回报 → 更新持仓 → 写入净值历史
```

此功能需要真实券商账户、API 申请资质和合规审查，不在当前迭代范围内。

## 十、明确不做的事

| 功能 | 不做的原因 |
|------|----------|
| Tick 级 / 分钟级实时行情 | 日线够用，分钟级需专业数据服务，成本高 |
| 强化学习 Agent | 信号基础数据不完善时做 RL 是在沙上建塔 |
| 多因子 Alpha 挖掘 | 单独研究项目，优先把现有 4 个 Agent 做可信 |
| 即时真实下单 | 需策略验证 6 个月以上，现在做是把错误决策自动化 |
| 复杂 OAuth / SSO 认证 | PIN + JWT 够用，过早引入增加维护成本 |

---

*核心迭代原则：先让信号有依据（v2.0），再让回测有意义（v3.0），再让风控真实有效（v4.0），最后才考虑自动执行（v8.0）。顺序不能颠倒——在信号不可信时做自动执行，等于把错误决策自动化。*
