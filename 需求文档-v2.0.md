## 需求一：仓位合计约束

### 1. 背景与问题

持仓权重合计理论上不可能超过 100%，但系统在三处均缺乏合计校验：

- `approvals.py modify_weights`：只校验单标的不超 `max_position_weight`，不校验全仓合计
- `portfolio.py apply_recommendations`：审批通过后直接覆盖持仓，无合计兜底
- `pipeline.py _make_recommendations`：调仓幅度用随机数，不感知当前总仓位水位

**复现步骤**：
1. 启动后端，POST `/api/v1/portfolio/` 三次，写入 `600519=0.18, 300750=0.15, 002594=0.60`，合计 93%
2. 直接向数据库插入一条 `approval_records`（status=pending，recommendations 含 600519）
3. PUT `/api/v1/approvals/{id}/modify`，body：`{"modified_weights": {"600519": 0.10}}`
4. 实际合计变为 103%，但系统返回 200 ← **Bug：应返回 422**

### 2. 接口定义

**修改接口**：`PUT /api/v1/approvals/{approval_id}/modify`（已有接口，新增校验逻辑）

请求 body（已有字段，无变化）：
```json
{
  "modified_weights": {"600519": 0.10},
  "comment": "调整权重"
}
```

新增校验逻辑（在现有单标的校验之后、写库之前加入）：
```
other_weight = sum(所有 portfolio_holdings 权重)
             - sum(modified_weights 中各标的在 portfolio_holdings 里的当前权重)
total = other_weight + sum(modified_weights.values())
if total > 1.001:
    raise HTTPException(422, detail=...)
```

422 响应体格式（数字保留一位小数）：
```json
{
  "detail": "修改后全仓合计108.0%超过100%，其他持仓已占93.0%，本次可用余量7.0%"
}
```

**新增接口**：无（只修改现有接口行为）

**pipeline 推荐量计算规则（修改 `_make_recommendations` 内部逻辑）**：
```
可用余量 = 1.0 - 其他所有持仓权重之和（不含当前标的）
buy  推荐权重 = min(当前权重 + min(0.10, 可用余量), max_position_weight)
sell 推荐权重 = max(当前权重 - 0.10, 0.0)
hold 推荐权重 = 当前权重（不变）
可用余量 == 0 时：buy 降级为 hold，low_data_warning 追加"当前仓位已满，无法加仓"
```

### 3. 错误情况

| 情况 | 状态码 | detail 内容 |
|------|--------|------------|
| 全仓合计超 100% | 422 | `"修改后全仓合计XX.X%超过100%，其他持仓已占XX.X%，本次可用余量XX.X%"` |
| 单标的超 max_position_weight | 422 | `"{sym}权重XX.X%超过上限XX.X%"` |
| 权重为负数 | 422 | `"{sym}权重不能为负数"` |
| 审批记录已处理（非 pending） | 409 | `"Approval already processed"` |
| approval_id 不存在 | 404 | `"Approval not found"` |

### 4. 实现注意

✅ `other_weight` 以 `portfolio_holdings` 表为准，排除 `modified_weights` 中已包含的标的
❌ 不要要求 `sum(modified_weights.values()) == 1.0`
   原因：`modified_weights` 只含本次审批标的，不是全部持仓，合计≠100%

✅ 422 时不调用 `apply_recommendations`，数据库不变
❌ 不要先写库再校验

✅ `apply_recommendations` 作为兜底：落库前计算落库后合计，超过 1.001 则抛异常回滚
❌ `apply_recommendations` 不需要对外返回 422，只需抛内部异常记日志

✅ pipeline 推荐量改为确定性计算（见接口定义第3段）
❌ 不要用 `random.uniform(0.03, 0.08)` 生成调仓幅度

- [ ] AC-1-04：持仓合计 93%，PUT modify 将某标的改为 7%（合计恰好 100%），返回 200

- [ ] AC-1-06：持仓合计 100%，pipeline 对任一标的输出 buy 方向时，推荐权重 == 当前权重（不增加）

## 需求二：图谱真实联通

### 1. 背景与问题

当前图谱模块 6 处使用随机数或占位值，图谱与分析流程完全割裂：

| 代码位置 | 问题 |
|---------|------|
| `services/graph.py:29` | `outcome_return = random.uniform(-0.08, 0.15)` |
| `services/graph.py:30` | `outcome_sharpe = random.uniform(-0.5, 3.0)` |
| `services/graph.py:17-20` | embedding 用 hash+random，与实际信号无关 |
| `pipeline.py:98` | `similar_cases: []` 始终为空，Agent 看不到历史 |
| `services/graph.py:56` | `similarity_trend` 用 `random.uniform(0.6, 0.95)` |
| `services/graph.py:73` | `edge_count = n-1` 占位 |

**复现步骤**：
1. GET `/api/v1/graph/stats`，多次刷新，`avg_accuracy` 和 `similarity_trend` 每次返回不同值 ← **Bug：统计数据应稳定**
2. 连续触发两次相同标的分析，GET `/api/v1/graph/nodes` 查看两个节点的 `embedding`，两者余弦相似度趋近于 0（因为是随机值）← **Bug：同条件分析应高度相似**
3. 审批通过一笔 buy，等待5个交易日，GET `/api/v1/graph/nodes`，`outcome_settled` 仍为 False ← **Bug：应自动结算**

### 2. 接口定义

**修改接口**：`GET /api/v1/graph/stats`（现有接口，返回值改为真实计算）

`avg_accuracy` 改为：`outcome_settled=True` 节点中 `outcome_direction_correct=True` 的占比
`similarity_trend` 改为：按月真实余弦相似度均值

**修改接口**：`POST /api/v1/graph/settle`（现有接口，触发真实结算逻辑）

**新增内部逻辑**（不新增对外 API）：
- `add_graph_node` 调用时：用 24 维特征向量替换随机 embedding
- `run_decision_pipeline` 入参新增 `graph_querier` 可选回调，由 `decisions.py` 传入
- `BaseAgent` system prompt 末尾：有已结算案例时追加历史参考段落

**GraphNode 新增字段**（已在 models.py 实现，确认以下字段存在）：
```
outcome_settled: bool = False
outcome_settled_at: datetime | None
outcome_direction_correct: bool | None
settle_price: float | None
approve_price: float | None
```

结算逻辑：
```
outcome_return = (settle_price - approve_price) / approve_price
direction_correct:
  buy  且 outcome_return > 0  → True
  sell 且 outcome_return < 0  → True
  hold                         → True
  其余                         → False
```

### 3. 错误情况

| 情况 | 处理方式 |
|------|---------|
| AKShare 拉取结算价格失败 | 跳过该节点，不标记 settled，下次重试，记录 warning 日志 |
| 图谱节点 embedding 计算时 signals 为空 | 返回全零向量，不报错 |
| graph_querier 查询图谱超时或异常 | pipeline 继续执行，similar_cases 降级为 `[]` |
| 余弦相似度计算时向量全为零 | 相似度返回 0.0，不除零报错 |

### 4. 实现注意

✅ `graph_querier` 作为可选参数传入 `run_decision_pipeline`，pipeline 本身不持有 db session
❌ 不要在 pipeline 内部直接 import 并调用数据库查询函数（会引入循环依赖）

✅ 相似案例注入 prompt 只取 `outcome_settled=True` 的节点
❌ 不要把未结算节点也注入 prompt（outcome 未知，参考价值为零）

✅ embedding 构建时对缺失字段用 0.0 填充，保持向量维度固定为 24
❌ 不要因为某个 Agent 信号缺失就跳过 embedding 构建或降低向量维度

✅ `get_graph_stats` 中 `avg_accuracy` 分母只计算已结算节点数量，并在返回值中标注 `settled_count`
❌ 不要把未结算节点计入准确率分母（会导致准确率被稀释）

✅ AKShare 价格查询用 `asyncio.to_thread()` 包裹
❌ 不要在 async 函数中直接调用 AKShare 同步接口

### 5. AC 验收条件

- [ ] AC-2-02：两个完全不同标的不同信号的节点，余弦相似度 < 0.50
- [ ] AC-2-03：审批通过后，图谱节点的 `approve_price` 字段被写入（非 None、非 0）
- [ ] AC-2-04：调用 POST `/graph/settle` 时，mock 结算价格 = 审批价格 * 1.05（buy），节点 `outcome_settled=True`，`outcome_return ≈ 0.05`，`outcome_direction_correct=True`
- [ ] AC-2-05：多次 GET `/graph/stats`，`avg_accuracy` 返回值相同（非随机）
- [ ] AC-2-06：图谱有 5 条已结算节点时触发分析，Agent 返回的 `reasoning` 字段包含历史案例描述文字
- [ ] AC-2-07：图谱无已结算节点时触发分析，`reasoning` 字段不包含"历史案例"或"相似案例"字样

## 需求三：Agent 权重管理基础设施

### 1. 背景与问题

当前 4 个 Agent 权重均硬编码为 0.25（`agents/base.py` 各子类的 `weight` 属性），无法根据历史表现调整。`AgentWeightConfig` 表和 `/api/v1/weights` 路由已在 v2.0 实现，但存在以下问题：

- pipeline 执行时仍使用硬编码权重，未读取数据库
- 权重优化逻辑（滑动窗口准确率）未实现，`rl_status` 返回占位数据
- 前端权重页面（`/weights`）在导航栏中无入口，用户无法访问

**复现步骤**：
1. PUT `/api/v1/weights/technical`，body：`{"weight": 0.40}`，返回 200
2. 触发一次分析，查看分析结果中 technical Agent 实际使用的权重
3. 实际权重仍为 0.25 ← **Bug：应读取数据库中的 0.40**

另：直接访问 `http://localhost:5173/weights` 可到达权重页，但导航栏无此入口。

### 2. 接口定义

**现有接口**（确认已实现，本需求补全逻辑）：

`GET /api/v1/weights/` — 返回 4 个 Agent 当前权重
```json
[
  {"agent_type": "technical", "weight": 0.25, "weight_source": "default", "locked": false},
  ...
]
```

`PUT /api/v1/weights/{agent_type}` — 手动更新权重
```json
{"weight": 0.40}
```
响应：更新后的 AgentWeightConfig 对象

`POST /api/v1/weights/{agent_type}/lock` — 切换锁定状态
```json
{"locked": true}
```

`GET /api/v1/weights/rl/status` — 返回优化状态
```json
{
  "settled_count": 12,
  "required_count": 60,
  "progress_pct": 20.0,
  "last_trained_at": null,
  "model_version": null,
  "accuracy_by_agent": {
    "technical": 0.67, "fundamental": 0.50, "news": 0.33, "sentiment": 0.58
  }
}
```

`POST /api/v1/weights/optimize` — 手动触发基于准确率的权重优化（已结算 ≥ 10 条时生效）

**Pipeline 读取权重**（修改 `pipeline.py`）：
```python
# run_decision_pipeline 开头，从数据库加载权重
weight_cfg = await load_agent_weights(db_session)
for agent in agents:
    agent.weight = weight_cfg.get(agent.agent_type, 0.25)
```

### 3. 错误情况

| 情况 | 状态码 | detail |
|------|--------|--------|
| agent_type 不在四个合法值内 | 422 | `"agent_type must be one of: technical, fundamental, news, sentiment"` |
| 权重值不在 [0.05, 0.45] 范围内 | 422 | `"weight must be between 0.05 and 0.45"` |
| 对 locked=True 的 Agent 更新权重 | 409 | `"Agent {type} is locked, unlock it first"` |
| 已结算数量 < 10 时调用 /optimize | 400 | `"Not enough settled decisions (need 10, have N)"` |

### 4. 实现注意

✅ `pipeline.py` 的 `run_decision_pipeline` 接收 `db_session` 参数，用它查 `AgentWeightConfig` 表
❌ 不要在 pipeline 内部创建新的数据库 session（已有 session 从 router 传入）

✅ `load_agent_weights` 返回 dict，缺失的 agent_type 用默认值 0.25 兜底
❌ 不要在 agent_type 不存在时报错中断 pipeline

✅ 权重优化（`/optimize`）使用滑动窗口准确率：取最近 N 条已结算节点各 Agent 的方向准确率，归一化后写入数据库
❌ 本版本不实现 PPO/RL 训练（需要 gymnasium），只做准确率统计优化

✅ 导航栏入口：在 `frontend/src/components/` 的侧边栏组件中，权重管理入口放在「规则」菜单项旁边
❌ 不要新建侧边栏组件，找到现有 sidebar/nav 组件修改

### 5. AC 验收条件

- [ ] AC-3-01：PUT `/weights/technical` 将权重改为 0.40，触发分析，technical Agent 实际使用权重 == 0.40（在 agent_signals 响应字段中可见）
- [ ] AC-3-02：PUT `/weights/technical` 权重改为 0.50（超上限），返回 422
- [ ] AC-3-03：POST `/weights/technical/lock`（locked=true），再 PUT `/weights/technical`，返回 409
- [ ] AC-3-04：GET `/weights/rl/status`，`settled_count` 与数据库中 `outcome_settled=True` 的节点数一致
- [ ] AC-3-05：已结算节点 < 10 时调用 POST `/weights/optimize`，返回 400
- [ ] AC-3-06：已结算节点 ≥ 10 时调用 POST `/weights/optimize`，数据库中 `weight_source` 变为 `"accuracy_based"`
